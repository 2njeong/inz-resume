## 같은 Wi-Fi = 같은 사설 네트워크

Wi-Fi 공유기(라우터)는 연결된 각 기기에 사설 IP를 할당한다.

```
공유기 (192.168.0.1)
  ├── 내 노트북: 192.168.0.108
  ├── 팀원 노트북: 192.168.0.112
  └── 팀원 핸드폰: 192.168.0.115
```

같은 공유기에 연결된 기기들은 `192.168.0.x` 대역의 사설 IP로 서로 직접 통신할 수 있다.

## 개발 서버가 외부 접근을 허용하는 원리

`npm run dev`를 실행하면 내부적으로 다음과 같은 코드가 실행된다:

```js
server.listen(3001, '0.0.0.0');
```

`0.0.0.0`은 "모든 네트워크 인터페이스에서 요청을 받겠다"는 의미다. 즉:

- `localhost` (127.0.0.1) -- 자기 자신의 요청
- `192.168.0.108` -- 같은 네트워크의 다른 기기 요청
- 기타 모든 네트워크 인터페이스

모두에서 들어오는 요청을 수락한다.

팀원은 브라우저에서 `http://192.168.0.108:3001`로 접속하면 내 로컬 서버의 응답을 받을 수 있다.

## localhost에 바인딩하면 외부 접근 불가

```js
server.listen(3001, '127.0.0.1');
```

`127.0.0.1` (localhost)에 바인딩하면 자기 자신의 요청만 수락한다. 같은 Wi-Fi의 다른 기기에서 접속하면 연결이 거부된다.

## 프레임워크별 설정 방법

### Vite

```bash
vite --host
# 또는
vite --host 0.0.0.0
```

```ts
// vite.config.ts
export default defineConfig({
  server: {
    host: true, // 0.0.0.0에 바인딩
  },
});
```

### Next.js

```bash
HOST=0.0.0.0 next dev
```

### Create React App

```bash
HOST=0.0.0.0 react-scripts start
```

### Express

```js
app.listen(3000, '0.0.0.0', () => {
  console.log('Listening on all interfaces');
});
```

## 방화벽이 차단할 수 있다

OS 방화벽이 특정 포트의 외부 접근을 차단할 수 있다. macOS에서 개발 서버를 처음 실행하면 "네트워크 연결을 허용하시겠습니까?" 팝업이 뜨는 이유가 이것이다.

방화벽에서 해당 포트를 허용해야 팀원이 접근할 수 있다.

## VPN이 연결되면 안 될 수 있다

VPN은 네트워크 트래픽을 VPN 서버로 터널링한다. VPN에 연결된 상태에서는 같은 Wi-Fi에 있더라도 트래픽이 VPN 네트워크를 경유하기 때문에 사설 IP 직접 통신이 불가능할 수 있다.

로컬 서버를 팀원과 공유해야 할 때는 VPN을 일시적으로 끄거나, split tunneling 설정을 확인해야 한다.
