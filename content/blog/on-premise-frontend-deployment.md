## 들어가며

프론트엔드 개발자도 On-Premise 환경에 배포할 때 네트워크 기초 개념을 알아야 한다. 10가지 핵심 개념을 정리한다.

## 1. VM vs 서버 프로그램 vs 프로세스

- **VM (가상 머신)**: 물리 서버 위에 가상화된 독립 컴퓨터. 자체 OS를 가진다.
- **서버 프로그램**: VM 위에서 실행되는 소프트웨어 (Nginx, Node.js, Tomcat 등).
- **프로세스**: 서버 프로그램이 실행되면 생기는 실체. 특정 포트를 점유한다.

하나의 VM에서 여러 서버 프로그램을 실행할 수 있고, 각각 다른 포트를 사용한다.

## 2. IP:Port = 1:N 관계

하나의 IP 주소에 여러 포트가 존재한다. 포트는 0~65535 범위이며, 각 포트에 하나의 프로세스가 바인딩된다.

```
VM (IP: 10.0.1.100)
  ├── :80   → Nginx
  ├── :3000 → Node.js (프론트엔드)
  └── :8080 → Spring Boot (백엔드)
```

## 3. Origin = Protocol + Host + Port

Origin은 **브라우저 보안 개념**이다. 서버 간 통신에서는 Origin이라는 개념이 없다.

```
https://example.com:443
  ├── protocol: https
  ├── host: example.com
  └── port: 443

http://example.com:80  ← 다른 Origin (protocol이 다름)
https://api.example.com:443  ← 다른 Origin (host가 다름)
```

CORS(Cross-Origin Resource Sharing)는 브라우저가 강제하는 정책이다. 서버에서 서버로의 요청에는 CORS가 적용되지 않는다.

## 4. 방화벽은 IP 단위로 제어한다

방화벽은 **IP-to-IP** 트래픽을 제어한다. "포트를 연다"는 표현은 편의상 쓰이지만, 실제로는 "특정 IP(또는 IP 대역)에서 이 서버의 특정 포트로의 접근을 허용한다"는 의미다.

```
규칙 예시:
  허용: 203.0.113.0/24 → 10.0.1.100:443
  차단: 0.0.0.0/0 → 10.0.1.100:8080
```

## 5. 같은 VM: localhost 통신은 방화벽을 우회한다

같은 VM 내에서 localhost(127.0.0.1) 또는 loopback 인터페이스를 통한 통신은 네트워크 스택을 거치지 않는다. 따라서 방화벽 규칙의 영향을 받지 않는다.

```
같은 VM 내:
  Nginx(:80) → localhost:8080 (Spring Boot)
  → 방화벽 무관, 항상 통신 가능
```

## 6. Reverse Proxy: 단일 진입점

Reverse proxy(주로 Nginx)는 외부 요청을 받아 내부 서비스로 라우팅한다. 외부에는 443 포트 하나만 노출한다.

```
인터넷 → :443 (Nginx)
            ├── /          → localhost:3000 (프론트엔드)
            ├── /api       → localhost:8080 (백엔드)
            └── /admin     → localhost:9090 (관리 도구)
```

## 7. Reverse Proxy 유무에 따른 차이

### Reverse Proxy 있을 때

```
브라우저 → https://example.com/api/users
        → Nginx(:443) → localhost:8080/users
```

- 모든 요청이 같은 Origin을 통해 들어오므로 **CORS 설정 불필요**
- 방화벽에서 443만 열면 됨
- SSL/TLS를 Nginx에서 한 번만 처리
- 인증/로깅을 한 곳에서 관리

### Reverse Proxy 없을 때

```
브라우저 → https://example.com:3000  (프론트엔드)
브라우저 → https://example.com:8080  (백엔드 직접)
```

- 포트가 다르므로 **CORS 설정 필요**
- 방화벽에서 3000, 8080 등 여러 포트를 열어야 함
- 각 서비스가 개별적으로 SSL 처리
- 인증/로깅이 분산됨

## 8. "포트를 연다" vs "방화벽에서 허용한다"

이 두 가지는 다른 개념이다.

- **포트를 연다**: 서버 프로그램이 해당 포트에서 listen하기 시작한다 (`server.listen(3000)`).
- **방화벽에서 허용한다**: 외부 IP에서 해당 포트로의 네트워크 접근을 허용한다.

서버 프로그램이 포트를 열어도 방화벽에서 차단하면 외부에서 접근할 수 없다. 반대로 방화벽에서 허용해도 해당 포트에서 listen하는 프로세스가 없으면 연결이 거부된다.

## 9. 고객 네트워크 허용 목록(Allowlist)

On-Premise 고객이 외부 서비스에 접근해야 할 때, 고객사 방화벽에서 우리 서버의 IP를 allowlist에 추가해야 한다.

```
고객사 방화벽:
  허용: 내부 네트워크 → 203.0.113.50:443 (우리 서비스)
```

이때 고객에게 전달해야 하는 정보:
- 서버 IP 주소 (또는 도메인)
- 사용하는 포트 번호
- 프로토콜 (TCP/UDP)

## 10. 정리: Proxy O vs Proxy X

| 항목 | Proxy 있음 | Proxy 없음 |
|------|-----------|-----------|
| 백엔드가 대화하는 상대 | Proxy만 (localhost) | 고객 브라우저 직접 |
| CORS 설정 | 불필요 (같은 Origin) | 필요 (다른 포트 = 다른 Origin) |
| 방화벽 관리 | 443 하나만 개방 | 서비스별 포트 개방 |
| SSL 관리 | Proxy에서 일괄 처리 | 서비스별 개별 처리 |
| 인증/로깅 | 한 곳에서 중앙 관리 | 서비스별 분산 관리 |
| 복잡도 | 낮음 | 높음 |

Reverse proxy를 사용하면 백엔드는 proxy하고만 통신하면 되므로 구조가 단순해진다. 사용하지 않으면 백엔드가 고객 브라우저를 직접 상대해야 하므로 복잡도가 높아진다.
